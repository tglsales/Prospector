TGL PROSPECTOR
==============

Chrome extension for managing LinkedIn outbound sequences. Scrapes profiles
automatically, generates personalized messages via an n8n webhook, and tracks
each prospect through a 3-touch sequence.


INSTALLATION
------------

1. Go to chrome://extensions
2. Enable Developer mode (top right)
3. Click Load unpacked and select this directory
4. Navigate to any LinkedIn profile — the side panel opens automatically


CONFIGURATION
-------------

Before generating messages, replace the hardcoded webhook URL at the top of
sidebar.js with your own:

    const WEBHOOK_URL = 'http://localhost:5678/webhook/your-webhook-id';

The webhook receives a userMessage string and must return { message: string }.

Note for forks: this is the only value you need to change. No API keys are
hardcoded — the Anthropic key (if used directly) is stored at runtime in
chrome.storage.local and never touches the codebase.


HOW IT WORKS
------------

Profile scraping
  content.js runs on every linkedin.com/in/* page. It waits until both the
  page title and the h1 agree on the same name (preventing stale DOM reads
  during SPA navigation), then scrapes name, role, company, photo URL, and
  company logo URL. The profile is stored in chrome.storage.local as
  currentProfile.

Sequence flow
  Each prospect goes through 3 touches with a 7-day gap between each:

    Touch 1  J+0   Generated by n8n webhook (personalized with signal + examples)
    Touch 2  J+7   Fixed template (competitor signal example)
    Touch 3  J+14  Fixed template (job change ROI example)

  Touch 2 and 3 are intentionally static — they exist to close the loop,
  not to personalize further.

Message generation (Touch 1)
  The webhook payload includes:
    - Prospect name, role, company
    - Signal / context (entered when adding the prospect)
    - Previous sent messages (for follow-up touches)
    - Up to 10 recent human edits to generated messages (few-shot learning)
    - Language instruction (fr or en)

  The extension learns from your edits: whenever you modify a generated
  message before sending it, the diff is saved and fed back into future
  generation prompts.

Alarms
  When a step is marked as sent, a chrome.alarms entry is created for the
  next step 7 days later. On alarm fire, isDue is set to true on that step
  and a desktop notification appears.


UI OVERVIEW
-----------

List view
  - All prospects sorted by urgency (follow-up due > to contact > sent > replied > meeting)
  - Red badge on prospects with a due follow-up
  - Search by name or company
  - Statistics panel (contacted, replied, meetings, reply rate, conversion rate)

Detail view
  - Avatar with company logo badge
  - Editable role and company (pencil button in header)
  - Sequence timeline: each step shows its status (active, due, sent, locked)
  - Per-step message generation with FR / EN flag buttons — language preference is remembered
  - Copy button for each step
  - Mark as sent -> triggers next step alarm
  - Replied / Meeting Booked buttons
  - Notes field (auto-saved on blur)


FILE STRUCTURE
--------------

  manifest.json   MV3 manifest, permissions, content script config
  background.js   Service worker: panel behavior, n8n proxy, alarm handler
  content.js      LinkedIn scraper, SPA navigation watcher
  sidebar.html    Side panel markup (list / detail / stats views)
  sidebar.css     All styles
  sidebar.js      All UI logic: rendering, actions, storage helpers
  icons/          Extension icons (16, 48, 128px)


DATA MODEL
----------

Prospects are stored in chrome.storage.local under the "prospects" key,
keyed by LinkedIn path (e.g. /in/john-doe):

  {
    id            LinkedIn path (/in/john-doe)
    firstName, lastName, fullName
    role, company
    photoUrl, companyLogoUrl    raw LinkedIn CDN URLs
    signal                      context entered at add time
    language                    last used generation language (fr or en)
    notes                       free-form notes
    addedAt, repliedAt, meetingAt
    currentStep                 index of the next step to send
    steps: [
      { number, type, delay, content, generatedContent, sentAt, isDue }
    ]
  }

Few-shot examples (generated vs. sent pairs) are stored separately under
the "examples" key, capped at 10 entries.
